# AGENTS.md — Contributor Guide for AI Agents

This file provides guidance for AI agents (and human contributors) working on the **GitHub Copilot Atlas** repository. It defines conventions, workflows, and standards that all agents must follow to contribute consistently and safely.

---

## Plan Directory

```
plans/
```

Atlas and Prometheus store all plan files (`.md`) in `plans/` at the workspace root. Create this directory if it does not yet exist.

---

## Repository Overview

GitHub Copilot Atlas is a **Markdown-only toolkit** — there is no application code, build step, or test runner. Every artifact is a Markdown or shell file:

| Directory | Purpose |
|-----------|---------|
| `agents/` | Custom agent definitions (`.agent.md` files with YAML frontmatter) |
| `skills/` | Agent skills (a `SKILL.md` and optional helper scripts per skill) |
| `instructions/` | Copilot workspace instructions template |
| `hooks/` | Documentation and examples for Copilot CLI lifecycle hooks |
| `plans/` | Plan files generated by Prometheus/Atlas (gitignored or committed as needed) |

There is no `package.json`, `Makefile`, or test suite. Validation is done by reading and reviewing the files themselves.

## Branch Naming

| Scenario | Pattern | Example |
|----------|---------|---------|
| Copilot automation | `copilot/<short-description>` | `copilot/add-security-auditor-agent` |
| New feature | `feat/<short-description>` | `feat/add-database-agent` |
| Bug fix | `fix/<short-description>` | `fix/atlas-plan-directory-lookup` |
| Documentation | `docs/<short-description>` | `docs/update-install-guide` |
| Chore / housekeeping | `chore/<short-description>` | `chore/clean-up-agent-frontmatter` |

Rules:
- Use **kebab-case** (lowercase, hyphen-separated).
- Keep descriptions short (3–5 words).
- Never use `main` or `master` as a working branch.

## Commit Messages

Follow the **Conventional Commits** format with an imperative-mood subject line:

```
<type>: <short description (max 50 chars)>

- Bullet summarising change 1
- Bullet summarising change 2

Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

### Types

| Type | When to use |
|------|------------|
| `feat` | New agent, skill, instruction block, or hook example |
| `fix` | Correcting a bug in an agent's logic or a broken script |
| `docs` | README, AGENTS.md, install.md, hooks/README.md changes |
| `chore` | Maintenance (renaming files, updating YAML frontmatter versions) |
| `refactor` | Restructuring agent instructions without changing behaviour |
| `test` | Adding or updating validation steps or example outputs |

### Rules

- Subject line uses **imperative mood**: `Add` not `Added`, `Fix` not `Fixed`.
- Keep the subject line under **50 characters**.
- One **logical change per commit** — don't bundle unrelated edits.
- Always include the `Co-authored-by` trailer for Copilot-authored commits.
- Never commit secrets, credentials, or `.env` files.

### Examples

```
feat: add Security-Auditor subagent

- Define agent with vulnerability review workflow
- Wire into Atlas subagent roster
- Add delegation note to Prometheus

Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

```
fix: correct mcp-introspect.sh timeout default

- Change MCP_INTROSPECT_TIMEOUT default from 5s to 10s
- Match documented default in SKILL.md

Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

## Pull Request Conventions

### Title

Use the same `<type>: <description>` format as the commit subject:

```
feat: add Security-Auditor subagent for vulnerability review
```

### Description

Structure the PR body as follows:

```markdown
## Summary
<!-- One paragraph: what changed and why. -->

## Changes
- `agents/Security-Auditor-subagent.agent.md` — new agent definition
- `agents/Atlas.agent.md` — added Security-Auditor to subagent roster
- `agents/Prometheus.agent.md` — added delegation note

## Testing / Validation
<!-- How you verified the changes (manual review, smoke test, etc.). -->

## Notes
<!-- Optional: open questions, known limitations, follow-up issues. -->
```

### Rules

- Target the **`main`** branch.
- Keep PRs focused — one feature or fix per PR.
- Reference the related issue with `Closes #<issue-number>` in the description when applicable.
- All agent file changes must pass a self-review checklist (see [Agent Contribution Checklist](#agent-contribution-checklist)).

## Agent File Format

Every agent lives in `agents/` as a single `.agent.md` file. The file must begin with a YAML frontmatter block:

```yaml
---
description: 'One sentence describing what this agent does'
tools: ['tool1', 'tool2', 'tool3']
model: Claude Sonnet 4.5 (copilot)
---
```

### Required frontmatter fields

| Field | Type | Description |
|-------|------|-------------|
| `description` | string | Single-sentence summary shown in the Copilot UI |
| `tools` | list | MCP/built-in tools the agent is permitted to use |
| `model` | string | LLM to use (e.g., `Claude Sonnet 4.5 (copilot)`, `Gemini 3 Flash (Preview)`) |

### Optional frontmatter fields

| Field | When to include |
|-------|----------------|
| `agents` | List of subagents this agent can invoke (use `["*"]` for all) |
| `argument-hint` | Short description of what input the agent expects |
| `handoffs` | Labelled transitions to other agents (for Prometheus-style handoffs) |

### Body conventions

- Start the body with a clear **role statement**: `You are a [ROLE] called [Name].`
- Use `##` sections to separate major workflow phases.
- Use fenced code blocks (` ```markdown `) for output templates.
- Use `<xml-tag>` wrappers (e.g., `<workflow>`, `<stopping_rules>`) for structured sections that agents must parse.
- Keep instructions prescriptive and unambiguous — agents act on them literally.
- For subagents, always specify the **output contract** (what format to return results in).
- For primary agents, always specify **mandatory pause points** (where user approval is required).

## Adding a New Agent

1. **Create the file**: `agents/<AgentName>-subagent.agent.md` for subagents, `agents/<AgentName>.agent.md` for primary agents.
2. **Write the frontmatter** following the format above.
3. **Write the body** following body conventions.
4. **Wire it into the ecosystem**:
   - Add it to Atlas's `agents:` frontmatter list and the `<subagent_instructions>` section.
   - Add a delegation note to Prometheus's research delegation description if it's a research-capable agent.
5. **Self-review** using the checklist below.

---

## Adding a New Skill

1. **Create the directory**: `skills/<skill-name>/`
2. **Create `SKILL.md`**: describe invocation (`/skill-name`), prerequisites, step-by-step process, and expected output.
3. **Add helper scripts** (optional): shell scripts must be POSIX-compatible and include a usage comment at the top.
4. **Update `README.md`** with the new skill's name, invocation command, and description.

## Agent Contribution Checklist

Before opening a PR that modifies an agent file, verify each item:

- [ ] Frontmatter is valid YAML (no tabs, correct quoting).
- [ ] `description` field is present and a single sentence.
- [ ] `tools` list contains only tools the agent genuinely needs.
- [ ] `model` is one of the approved models in use by the repo.
- [ ] Role statement is clear in the first line of the body.
- [ ] Workflow sections are clearly delineated with `##` headers.
- [ ] Output contracts are defined for subagents (what the parent agent can expect).
- [ ] Mandatory pause/stop points are explicit for primary agents.
- [ ] No hardcoded paths, credentials, or environment-specific assumptions.
- [ ] The agent is wired into Atlas and/or Prometheus if it participates in the orchestration cycle.
- [ ] `README.md` is updated if the agent is new (add a row to the agents table).

## Subagent Delegation Patterns

When writing agent instructions that involve delegation, follow these patterns:

| Scenario | Pattern |
|----------|---------|
| Task touches > 5 files | Delegate to Explorer-subagent first for file discovery |
| Task spans > 2 subsystems | Invoke Oracle-subagent once per subsystem, in parallel |
| Task involves UI/frontend | Delegate to Frontend-Engineer-subagent |
| Implementation phase | Delegate to Sisyphus-subagent (one per disjoint work unit) |
| Code review | Delegate to Code-Review-subagent after each implementation phase |
| Deep research needed | Delegate to Oracle-subagent; Explorer for breadth, Oracle for depth |

**Maximum parallel subagents**: 10 per phase.  
**Context rule**: If a step requires reading > 1 000 tokens of file content, delegate it to a subagent.

## Style & Language Conventions

- All prose in agent files, skills, and documentation is written in **clear, direct English**.
- Use **title case** for agent names (`Atlas`, `Prometheus`, `Sisyphus`).
- Use **kebab-case** for filenames: `my-new-agent.agent.md`, `mcp-introspect.sh`.
- Use **`backticks`** for file names, paths, commands, and code inline.
- Use **bold** to highlight key terms, warnings, and mandatory steps.
- Prefer bullet lists and tables over long prose paragraphs.
- Do not add filler or hedge language ("perhaps", "you might want to", "feel free to").

---

## What Not to Do

- **Do not** commit application code, binaries, or build artefacts to this repository.
- **Do not** add new npm, pip, or system dependencies unless strictly necessary.
- **Do not** modify `install.sh` or `install.ps1` without testing the install flow on both macOS/Linux and Windows.
- **Do not** remove or rename existing agent files without updating every reference to them in Atlas, Prometheus, and the README.
- **Do not** introduce hardcoded user paths, machine-specific settings, or secrets.
- **Do not** merge to `main` without a PR and at least one approval.
